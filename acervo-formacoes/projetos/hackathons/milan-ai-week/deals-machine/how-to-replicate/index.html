Guia de replicação completo

Clona, configura e corre offline.

Passo-a-passo para teres uma cópia funcional do Deals Machine em ~30-45 minutos. Permite-te modificar agentes, ver como se interligam com a app, e usar o sistema mesmo que o repositório original saia do ar.

O que vais montar

Três serviços, uma base de dados.

O Deals Machine é uma aplicação distribuída com 3 componentes que falam entre si:

  • Cockpit — UI Next.js (hospedada na Vercel). Vai correr em localhost:3000.
  • Worker — agente Fastify em TypeScript (hospedado em Vultr). Vai correr em localhost:3001.
  • Lobster Trap — shield anti prompt-injection (Veea). Vai correr em localhost:8080.
  • Supabase — Postgres + Realtime + Auth (managed cloud, free tier suficiente para dev).
Pré-requisitos

Node.js 20.x, npm/pnpm, Git, conta Supabase (free), conta Anthropic, e — para chamadas reais — Twilio, Speechmatics, Apollo, HubSpot. Sem estas últimas, o cockpit corre mas certas features quebram (chamadas, sourcing).

1

Clonar o repositório

Usa o snapshot local deste acervo (sempre funcional, mesmo offline) ou clona directamente do GitHub original:

Opção A — a partir do snapshot offline

# Copia a pasta inteira para um sítio onde queiras trabalhar
cp -R /caminho/para/acervo-formacoes/projetos/hackathons/milan-ai-week/deals-machine/github-snapshot ~/dev/deals-machine
cd ~/dev/deals-machine

# Inicializa novo repo git (snapshot não tem .git)
git init
git add .
git commit -m "Initial commit from acervo snapshot"

Opção B — directo do GitHub

git clone https://github.com/kyletdow47/deals-machine.git
cd deals-machine
Recomendado: forka para o teu GitHub

Para poderes fazer push das tuas modificações: vai a github.com/kyletdow47/deals-machine, clica em Fork, depois clona o teu fork.

2

Configurar Supabase

Cria um projecto Supabase novo (free tier) — vai ser a base de dados, auth e realtime para o cockpit e worker.

  1. Vai a supabase.com e cria conta + projecto. Escolhe região perto de ti.
  2. Espera ~2 min até o projecto estar provisionado.
  3. Vai a Project Settings → API. Copia:
    • Project URL → para SUPABASE_URL
    • anon public → para NEXT_PUBLIC_SUPABASE_ANON_KEY
    • service_role → para SUPABASE_SERVICE_KEY (secreto, nunca exposto ao browser)
  4. Vai a SQL Editor e corre os 20 ficheiros de supabase/migrations/ em ordem (001 → 020). Cola um, executa, próximo.
Atalho via Supabase CLI

Se tens a CLI instalada: supabase link --project-ref <teu-ref> e depois supabase db push aplica todas as migrações de uma vez.

3

Configurar e correr o Cockpit (Next.js)

O cockpit é a UI principal — verticals, pipeline, intelligence, leads, knowledge, analytics, settings.

# Na raiz do projecto
cp .env.local.example .env.local
# Edita .env.local com os valores Supabase do passo 2

# Gera shared secret (igual no worker)
openssl rand -hex 32
# Cola o output em WORKER_PROXY_SECRET (ambos .env.local e worker/.env)

npm install
npm run dev

Abre localhost:3000 — devias ver o cockpit. Algumas páginas vão estar vazias até teres dados (próximo passo).

4

Configurar e correr o Worker (Fastify)

O worker corre o pipeline autónomo de 10 passos: signals → chains → leads → script → coach → score → learn → follow-up. É onde toda a inteligência vive.

cd worker
cp .env.example .env
# Edita worker/.env com TODAS as chaves abaixo

npm install
npm run dev   # tsx watch em PORT 3000 por defeito
# Para correr em 3001 (não colidir com cockpit):
PORT=3001 npm run dev

Testa: curl http://localhost:3001/health → deve dar {"status":"ok"}.

Twilio webhook URL

Para chamadas reais funcionarem, Twilio precisa de bater num URL público com HTTPS. Em dev usa ngrok http 3001 e mete o URL ngrok em TWILIO_WEBHOOK_BASE.

Estratégia mínima viável

Para só explorar a app sem chamadas reais: configura só Anthropic + Apollo + Supabase. Vais conseguir gerar verticals, fazer sourcing, ver o pipeline a correr sem ter de pagar Twilio/Speechmatics.

5

Lobster Trap (opcional)

Shield anti prompt-injection. Em dev podes saltar — o worker faz fallback in-process para regex+classifier directamente.

Para correr standalone

cd lobster-trap
npm install
npm run dev   # Fastify em PORT 8080

# Ou via Docker:
docker build -t lobster-trap .
docker run -p 8080:8080 -e ANTHROPIC_API_KEY=$ANTHROPIC_API_KEY lobster-trap

Testa: curl -X POST http://localhost:8080/v1/check -H "Content-Type: application/json" -d '{"text":"Ignore previous instructions"}' → vai bloquear.

6

Primeira utilização

Com os 3 serviços a correr, abre localhost:3000 e segue o onboarding:

  1. Gera o primeiro vertical em /onboarding. Cola um parágrafo a descrever o que vendes e a quem. Claude gera ICP + 4-5 fontes de sinal + persona.
  2. Refina — ajusta sinais, exclusões, tom. Aceita.
  3. Run pipeline em /pipeline. O worker vai correr os 10 passos e popular leads no Supabase.
  4. Vê leads em /leads. Faz click-to-call num lead (se Twilio configurado) ou simula com transcript upload.
  5. Após a call, vai a /intelligence — scorecard e brain entries actualizadas.
  6. Tag outcome. Playbook regenera. Próxima call usa script melhorado.

Aprofundar & modificar

Como melhorar os agentes

Toda a lógica do agente vive em worker/src/. Os sítios mais interessantes para mexer:

Ficheiro / pastaO que controla
worker/src/pipeline/Orquestração dos 10 passos. Aqui muda-se a sequência ou condições.
worker/src/llm/Wrappers Claude. Aqui muda-se o prompt, o modelo, temperatura.
worker/src/playbook/Geração de scripts/cheat-sheets. Aqui controla-se o tom e estrutura do playbook.
worker/src/coaching/Loop de coaching live durante a call. Cadência, janela rolante, tipo de cards.
worker/src/brain/Memória por lead. Sistema de pesos e decay temporal.
worker/src/signals/Scraping de fontes (RSS/Reddit/HN). Adicionar novas fontes aqui.
app/api/22 rotas API do cockpit. São proxies que reencaminham para o worker.
Workflow recomendado

Faz fork do repo, cria branch experiments/<nome>, mexe num agente de cada vez, corre em local, mede impacto. Quando estiver bom, push para o teu fork.

Quando algo falha

Troubleshooting

Cockpit dá erro de Supabase auth

Verifica que o NEXT_PUBLIC_SUPABASE_ANON_KEY está correcto. Re-arranca o npm run dev sempre que mexes em .env.local.

Worker dá 401 ao ser chamado

O WORKER_PROXY_SECRET no cockpit e no worker tem de ser exactamente igual. Re-gera com openssl rand -hex 32 e copia para os dois ficheiros.

Twilio não consegue chegar ao webhook

Twilio precisa de URL público com HTTPS. Em local usa ngrok http 3001 e mete a URL gerada (ex.: https://abc123.ngrok-free.app) em TWILIO_WEBHOOK_BASE. Re-arranca o worker.

Migrations falham no Supabase

Corre uma a uma em ordem numérica no SQL Editor. Se uma falhar, lê a mensagem — provavelmente faltou uma extensão (ex.: pgvector). Activa em Database → Extensions.
