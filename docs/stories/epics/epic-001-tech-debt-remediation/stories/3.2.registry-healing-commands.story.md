# Story 3.2: Comandos `--recompute-checksums` / `--detect-orphans` / `--rebuild` para o registry

**Epic:** epic-001-tech-debt-remediation
**Status:** Draft
**Wave:** 3
**TD source:** DS-002
**Severity:** MEDIUM (pré-requisito filesystem-first sustentável)
**Effort:** M-L (1-2 dias)

## Goal

Adicionar a `aiox doctor` (ou novo comando `aiox registry`) os subcommands de **healing** do `entity-registry.yaml` (821 entidades, 580 KB). Hoje `aiox doctor` é apenas diagnóstico, não healing. Edits manuais a esta escala são inviáveis sem tooling.

## Acceptance Criteria

- [ ] AC1: Existe `aiox registry --recompute-checksums` que recalcula todos os checksums declarados em `entity-registry.yaml` e actualiza o ficheiro
- [ ] AC2: Existe `aiox registry --detect-orphans` que reporta entidades sem ficheiro correspondente no FS e ficheiros sem entrada no registry (em ambas as direcções)
- [ ] AC3: Existe `aiox registry --rebuild` que reconstrói o registry inteiro a partir dos ficheiros no FS (resultado deve ser determinístico — repetir produz output idêntico)
- [ ] AC4: Os comandos suportam flag `--dry-run` (mostra mudanças sem aplicar)
- [ ] AC5: Existe campo `metadata.lastRebuiltBy` (ou `lastUpdatedAt`) no `entity-registry.yaml` para audit trail
- [ ] AC6: `aiox registry --rebuild` em ambiente limpo produz registry idêntico (byte-a-byte ou semanticamente idêntico) ao actual — teste de determinismo
- [ ] AC7: Documentação curta em `.aiox-core/core/registry/README.md` (ou similar) descrevendo cada subcommand
- [ ] AC8: Comandos integram com schemas de Story 3.1 (validação após rebuild)

## Tasks

- [ ] Task 1: Auditar estrutura actual de `aiox doctor` e decidir: estender vs novo comando `aiox registry`
- [ ] Task 2: Implementar `--recompute-checksums`
- [ ] Task 3: Implementar `--detect-orphans` (lê FS + registry, compara)
- [ ] Task 4: Implementar `--rebuild` (scan FS, generate entries, write registry)
- [ ] Task 5: Adicionar `--dry-run` a todos
- [ ] Task 6: Adicionar campo `metadata.lastRebuiltBy` ao schema (coordena com Story 3.1)
- [ ] Task 7: Teste de determinismo: rebuild → compare bytes (ou tree estrutural) com original
- [ ] Task 8: Documentação
