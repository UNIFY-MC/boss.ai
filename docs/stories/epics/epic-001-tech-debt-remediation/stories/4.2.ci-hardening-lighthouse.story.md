# Story 4.2: Headers pós-deploy + link checker + Lighthouse CI budget

**Epic:** epic-001-tech-debt-remediation
**Status:** Draft
**Wave:** 4
**TD source:** TD-016 (+ TD-026 absorvido)
**Severity:** LOW (mas alto ROI — CI passa de "verde por sintaxe" a "verde por comportamento")
**Effort:** S (2-3h)

## Goal

Endurecer o CI com 3 verificações pós-deploy que cobrem comportamento + performance:
1. **Headers pós-deploy** — `curl -I` ao URL deployed da Vercel valida que `Content-Security-Policy`, `Strict-Transport-Security`, `Permissions-Policy` etc. estão presentes
2. **Link checker** — `lychee` (ou equivalente) corre nos HTMLs do acervo e detecta links partidos
3. **Lighthouse CI** com performance budget JSON: LCP <2.5s mobile 3G, CLS <0.1, INP <200ms, asset máx 100 KB

## Acceptance Criteria

- [ ] AC1: CI step (job separado ou step adicional) corre `curl -I <vercel-url>` após deploy e valida presença de headers críticos (CSP, HSTS, X-Frame-Options, X-Content-Type-Options, Referrer-Policy, Permissions-Policy)
- [ ] AC2: CI step corre `lychee` (ou link-checker equivalente) em todos os HTMLs de `acervo-formacoes/` e falha em caso de link partido (excluindo wildcards aceites como `https://*.example`)
- [ ] AC3: CI step corre Lighthouse CI com `lighthouserc.json` definindo budget:
  - LCP < 2500ms
  - CLS < 0.1
  - INP < 200ms
  - Asset máximo individual < 100 KB
  - A11y score >= 95
- [ ] AC4: Budget aplicado a pelo menos 3 páginas representativas do acervo (uma por sub-design-system)
- [ ] AC5: Falha em qualquer um dos 3 checks bloqueia merge (job required)
- [ ] AC6: Documentação curta em `.github/workflows/README.md` ou commit message explicando os 3 novos jobs

## Tasks

- [ ] Task 4: Adicionar 3 jobs (ou steps) ao workflow CI
- [ ] Task 5: Testar cada um deliberadamente (introduzir falha → CI bloqueia → corrigir)
